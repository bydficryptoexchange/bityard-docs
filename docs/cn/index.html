    HTTP返回代碼

    • HTTP 4XX錯誤碼用於指示錯誤的請求內容、行為、格式。
    • HTTP 403錯誤碼表示違反WAF限制(Web應用程序防火牆)。
    • HTTP 429錯誤碼表示警告訪問頻次超限,即將被封IP
    • HTTP 418表示收到429後繼續訪問,於是被封了。
    • HTTP 5XX錯誤碼用於指示BitYard服務側的問題。
    • HTTP 503表示三種可能：
      1. 如果返回內容裡包含了報錯信息"Unknown error, please check your request or try again later."，則表示API服務端已經向業務核心提交了請求但未能獲取響應，特別需要注意的是其不代表請求失敗，而是未知。很可能已經得到了執行，也有可能執行失敗，需要做進一步確認。
      2. 如果返回內容裡包含了報錯信息"Service Unavailable."，則表示本次API請求失敗。這種情況下可能是服務暫不可用，您需要稍後重試。
      3. 如果返回內容裡包含了報錯信息"Internal error; unable to process your request. Please try again."，則表示本次API請求失敗。這種情況下您如果需要的話可以選擇立即重試。

    訪問限制

    • 在/api/dapi/v1/exchangeInfo接口中rateLimits數組裡包含有REST接口(不限於本篇的REST接口)的訪問限制。包括帶權重的訪問頻次限制、下單速率限制。本篇枚舉定義章節有限制類型的進一步說明。
    • 違反上述任何一個訪問限制都會收到HTTP429,這是一個警告.
    請注意，若用戶被認定利用頻繁掛撤單且故意低效交易意圖發起攻擊行為，BitYard有權視具體情況進一步加強對其訪問限制。

    IP訪問限制

    • 每個路由都有一個"權重",該權重確定每個接口計數的請求數。較重的接口和對多個交易對進行操作的接口將具有較重的"權重"。
    • 收到429時,您有責任作為API退回而不向其發送更多的請求。
    • 如果屢次違反速率限制和/或在收到429後未能退回,將導致API的IP被禁(http狀態418)。
    • 頻繁違反限制,封禁時間會逐漸延長,對於重複違反者,將會被封從2分鐘到3天。
    • 訪問限制是基於IP的,而不是APIKey
    強烈建議您盡可能多地使用websocket消息獲取相應數據,既可以保障消息的及時性，也可以減少請求帶來的訪問限制壓力。

    下單頻率限制

    • 被拒絕或不成功的下單並不保證回報中包含以上頭內容。
    • 下單頻率限制是基於每個帳戶計數的。

    接口鑑權類型

    • 每個接口都有自己的鑑權類型,鑑權類型決定了訪問時應當進行何種鑑權
    • 如果需要API-key,應當在HTTP頭中以X-BITYARD-APIKEY字段傳遞
    • API-key與API-secret是大小寫敏感的
    • 可以在網頁用戶中心修改API-key所具有的權限,例如讀取帳戶信息、發送交易指令、發送提現指令
    鑑權類型 描述
    NONE 不需要鑑權的接口
    TRADE 需要有效的API-KEY和簽名
    USER 需要有效的API-KEY和簽名
    USER_DATA 需要有效的API-KEY和簽名
    USER_STREAM 需要有效的API-KEY
    MARKET_DATA 需要有效的API-KEY

    需要簽名的接口(TRADE與USER_DATA)

    • 調用這些接口時,除了接口本身所需的參數外,還需要傳遞sign即簽名參數。
    • 簽名使用HMAC SHA256算法.API-KEY所對應的API-Secret作為HMAC SHA256的密鑰,其他所有參數作為HMAC SHA256的操作對象,得到的輸出即為簽名。
    • 簽名大小寫不敏感。

    時間同步安全

    • 簽名接口均需要傳遞timestamp參數,其值應當是請求發送時刻的unix時間戳(毫秒)
    • 服務器收到請求時會判斷請求中的時間戳,如果是5000毫秒之前發出的,則請求會被認為無效。這個時間窗口值可以通過發送可選參數recvWindow來自定義。
    • 另外,如果服務器計算得出客戶端時間戳在服務器時間的‘未來’一秒以上,也會拒絕請求。

    邏輯偽代碼：

      if (timestamp < (serverTime + 1000) && (serverTime - timestamp) <= recvWindow) {
    // process request
  } else {
    // reject request
  }

    關於交易時效性互聯網狀況並不100%可靠,不可完全依賴,因此你的程序本地到BITYARD服務器的時延會有抖動.這是我們設置recvWindow的目的所在,如果你從事高頻交易,對交易時效性有較高的要求,可以靈活設置recvWindow以達到你的要求。

    不推薦使用5秒以上的recvWindow

    2. 開始緩存收到的更新。同一個價位,後收到的更新覆蓋前面的。
    3. 訪問Rest接口https://test-ws.bydtms.com/openapi/api/dapi/v1/depth?symbol=BTC-USD&limit=1000獲得一個1000檔的深度快照
    4. 將目前緩存到的信息中u{{<步骤3中获取到的快照中的}}lastUpdateId的部分丟棄(丟棄更早的信息,已經過期)。
    5. 將深度快照中的內容更新到本地orderbook副本中,並從websocket接收到的第一個u <= lastUpdateId 且 u >= lastUpdateId的event開始繼續更新本地副本。
    6. 將目前緩存到的信息中u{{<步骤3中获取到的快照中的}}lastUpdateId的部分丟棄(丟棄更早的信息,已經過期)。
    7. 將深度快照中的內容更新到本地orderbook副本中,並從websocket接收到的第一個u >= lastUpdateId的event開始繼續更新本地副本。
    8. 每一個新event的pu應該等於上一個event的u,否則可能出現了丟包,請從step3重新進行初始化。
    9. 每一個event中的掛單量代表這個價格目前的掛單量絕對值,而不是相對變化。
    10. 如果某個價格對應的掛單量為0,表示該價位的掛單已經撤單或者被吃,應該移除這個價位。

    POST /api/dapi/v1/order (HMAC SHA256)

    權重: 1

    參數:

    名稱 類型 是否必須 描述
    symbol STRING YES 交易對
    side ENUM YES 買賣方向SELL, BUY
    positionSide ENUM NO 持倉方向,單向持倉模式下非必填,默認且僅可填BOTH;在雙向持倉模式下必填,且僅可選擇LONG或SHORT
    type ENUM YES 訂單類型LIMIT, MARKET,STOP_MARKET, TAKE_PROFIT_MARKET
    quantity DECIMAL NO 下單數量,使用closePosition不支持此參數。
    price DECIMAL NO 委託價格
    newClientOrderId STRING NO 用戶自定義的訂單號
    stopPrice DECIMAL NO 觸發價,STOP_MARKET, TAKE_PROFIT_MARKET需要此參數
    timeInForce STRING NO 有效方法,GTC成交為止, IOC無法立即成交的部分就部分撤銷 FOK無法全部立即成交就撤銷
    closePosition STRING NO true, false；觸發後全部平倉,僅支持STOP_MARKET和TAKE_PROFIT_MARKET；不與quantity合用；自帶只平倉效果合用
    workingType ENUM NO stopPrice觸發類型:MARK_PRICE(標記價格),CONTRACT_PRICE(合約最新價).默認CONTRACT_PRICE
    recvWindow LONG NO
    timestamp LONG YES

    根據ordertype的不同,某些參數強制要求,具體如下:

    Type 強制要求的參數
    LIMIT timeInForce, quantity, price
    MARKET quantity
    STOP_MARKET, TAKE_PROFIT_MARKET stopPrice,workingType

    • 條件單的觸發必須:

      • STOP_MARKET止損單:
        • 買入:最新合約價格/標記價格高於等於觸發價stopPrice
        • 賣出:最新合約價格/標記價格低於等於觸發價stopPrice
      • TAKE_PROFIT_MARKET止盈單:
        • 買入:最新合約價格/標記價格低於等於觸發價stopPrice
        • 賣出:最新合約價格/標記價格高於等於觸發價stopPrice

    • STOP_MARKET, TAKE_PROFIT_MARKET配合closePosition=true:

      • 條件單觸發依照上述條件單觸發邏輯
      • 條件觸發後,平掉當時持有所有多頭倉位(若為賣單)或當時持有所有空頭倉位(若為買單)
      • 不支持此參數。quantity參數
      • 雙開模式下,LONG方向上不支持BUY; SHORT方向上不支持SELL

    Request Name <listenKey>@account

    帳戶更新事件的eventtype固定為ACCOUNT_UPDATE

    • 僅當帳戶信息有變動時,才會推送此事件。
    • 訂單狀態變化沒有引起帳戶和持倉變化的,不會推送此事件。

    • 每次推送的position信息,僅包含當前持倉不為0或逐倉倉位保證金不為0的symbolposition。

    • 字段"m"代表了事件推出的原因，包含了以下可能類型:

      • DEPOSIT
      • WITHDRAW
      • ORDER
      • ADJUSTMENT
      • INSURANCE_CLEAR
      • ADMIN_DEPOSIT
      • ADMIN_WITHDRAW
      • MARGIN_TRANSFER
      • MARGIN_TYPE_CHANGE
      • ASSET_TRANSFER

    • 字段"bc"代表了錢包餘額的改變量，即balancechange，但注意其不包含倉位盈虧及交易手續費。
